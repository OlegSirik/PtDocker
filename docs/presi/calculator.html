Почему традиционные страховые системы тормозят развитие бизнеса

Большинство страховых компаний продолжают использовать системы, в которых тарифная логика встроена непосредственно в программный код.

Любое изменение продукта требует:

  • подготовки требований;
  • участия аналитиков;
  • доработки программного обеспечения;
  • тестирования;
  • выпуска новой версии системы.

В результате даже небольшое изменение тарифа может занимать недели или месяцы.

Основные проблемы

🐢 Медленный вывод продуктов
Запуск нового продукта требует нескольких релизных циклов.
💰 Высокая стоимость изменений
Каждое изменение тарифа превращается в ИТ-проект.
🔗 Зависимость бизнеса от разработки
Бизнес-пользователи не могут самостоятельно управлять продуктом.
📋 Дублирование логики
Одинаковые коэффициенты и формулы копируются между продуктами.
Современная страховая компания должна иметь возможность изменять продукт так же быстро, как меняются потребности клиентов.

Configuration over Code

Rating Engine основан на принципе Configuration over Code. Вместо программирования правил расчета используются конфигурационные объекты.

Основные элементы модели

Input

Входные параметры договора. Примеры: возраст клиента, страховая сумма, регион, тип транспортного средства.

Rate Table

Тарифные таблицы. Используются для определения коэффициентов, тарифов и лимитов.

Formula

Формулы расчета. А также Constant (константы) и Variable (промежуточные и итоговые результаты).

Такой подход позволяет создавать сложные тарифные модели без изменения исходного кода платформы.

Тарифные таблицы

Rate Table является одним из ключевых компонентов системы. Тарифная таблица позволяет определять значение по одному или нескольким параметрам договора.

Примеры использования

  • территориальные коэффициенты

  • комиссии

Расчет премии по покрытиям

Rating Engine поддерживает детализированный расчет по каждому покрытию. Пример: DAMAGE, THEFT, ACCIDENT

Для каждого покрытия рассчитывается собственная премия.

co_DAMAGE_premium co_THEFT_premium co_ACCIDENT_premium

После расчета покрытий формируется итоговая премия договора.

Такой подход обеспечивает:

  • прозрачность расчета
  • объяснимость тарификации
  • гибкую настройку продуктов
  • поддержку пакетных продуктов
